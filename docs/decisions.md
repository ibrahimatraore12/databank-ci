# Journal des décisions de conception - dataBank CI Customer 360

> *[English version: [decisions_en.md](decisions_en.md)]*

**Auteur :** Ibrahima TRAORÉ - Analytics Engineer

Ce document liste les choix importants pris pendant le projet, pourquoi
ils ont été faits, et les autres options qui n'ont pas été retenues.

Format du tableau : Décision | Raison | Autre option écartée

| Décision | Raison | Autre option écartée |
|----------|--------|----------------------|
| Le score de désengagement est calculé en deux temps : des règles métier explicites en Étape 1, puis un essai de Machine Learning sur un indicateur approché en Étape 2 | Le fichier source ne contient aucun départ de client réellement observé (voir `docs/ml_problem_definition.md`). Présenter un score de Machine Learning comme une vérité certaine aurait été trompeur | Entraîner directement un modèle de Machine Learning, sans passer par une étape de règles métier |
| Génération d'un jeu de données synthétique (`is_synthetic=True`) pour faire passer le volume de 140 à environ 540 clients | Avec un seul critère simple, seulement 2 clients sont concernés (1,4 %). Même avec un critère plus large (25,0 %), l'échantillon reste petit sur 140 clients. Un échantillon de test plus grand est nécessaire pour comparer les modèles de façon fiable | Entraîner uniquement sur les 140 clients réels et accepter des résultats peu fiables |
| Le seuil retenu pour définir un client "à risque" est fixé à au moins 2 critères sur 4 (et non ajusté pour obtenir un taux précis à l'avance) | Mesuré sur les données réelles : 1 seul critère donne 70 % des clients concernés (beaucoup trop), 2 critères donnent 25,0 %, 3 critères donnent seulement 0,7 % (trop peu). Le seuil à 2 critères est le meilleur compromis, gardé tel quel plutôt qu'ajusté après coup pour viser un chiffre précis | Chercher un seuil ou une pondération qui donnerait artificiellement 12 à 15 % de clients concernés |
| Toutes les données générées artificiellement restent marquées `is_synthetic=True` dans les tables finales et dans le tableau de bord | Pour que personne ne puisse confondre une donnée générée avec une vraie donnée client | Mélanger silencieusement les données réelles et synthétiques |
| Le revenu généré par client (NBI) est estimé avec la formule standard UEMOA, et marqué `estimated_nbi_flag=True` | Le chiffre comptable réel du NBI n'existe pas dans le fichier source | Ne calculer aucun NBI du tout |
| Une même valeur de départ (`seed=42` et `random_state=42`) est utilisée partout : les tests dbt, la séparation des données pour le Machine Learning, la génération de données synthétiques | Pour que relancer le pipeline donne toujours exactement le même résultat | Laisser le hasard varier à chaque exécution |
| Aucun nom de colonne technique n'apparaît dans le tableau de bord : tout passe par le fichier `LABELS` (`dashboard/components/ui.py`) et les fichiers de traduction `i18n/*.json` | Le tableau de bord est destiné aux conseillers commerciaux, pas à des personnes techniques | Afficher directement les noms de colonnes SQL ou pandas |
| Le seuil de retard de paiement (`days_past_due`) est fixé à 15 jours pour corriger le statut d'un prêt dans `stg_loans.sql` | Observé pendant l'analyse exploratoire : les prêts réellement classés `Watchlist` ou `Delinquent` dépassent ce seuil, les prêts `Current` restent en dessous | Utiliser directement le statut du fichier source, sans le vérifier par rapport au retard réel |
| Les fichiers chargés par un utilisateur sont sauvegardés dans un espace de stockage Google Cloud Storage privé, rechargé au démarrage de chaque instance (`src/storage_sync.py`) | Cloud Run ne garde rien en mémoire de façon permanente : un fichier chargé depuis l'onglet Administration serait perdu au premier redémarrage sans cette sauvegarde externe. Détail dans `docs/architecture.md`, section 6 | Sauvegarder uniquement le fichier Excel source, ou utiliser une base de données gérée séparément |
| Une vérification de version du schéma (`config.DATA_SCHEMA_VERSION`) est faite avant tout téléchargement depuis l'espace de stockage GCS | Sans cette vérification, un futur changement de structure dans dbt pourrait être silencieusement écrasé au démarrage par une ancienne sauvegarde restaurée, provoquant des erreurs en chaîne sur toute requête utilisant la nouvelle colonne | Télécharger sans aucune vérification et accepter le risque d'incompatibilité |
| Le fichier `mlflow.db` est copié proprement grâce à la fonction `sqlite3.Connection.backup()` vers un fichier temporaire avant d'être envoyé sur GCS, jamais copié directement octet par octet | MLflow peut garder une connexion ouverte en tâche de fond ; une copie brute risquerait de capturer les données à un moment incohérent | Copier le fichier `.db` directement, sans passer par cette étape de sauvegarde propre |
| La page "Opportunités Commerciales" est construite à partir des taux d'acceptation réels observés dans l'historique, par type d'offre (`main_staging.stg_offers`), et non à partir d'une probabilité prédite pour chaque client | Aucun modèle de ce projet ne calcule une probabilité d'acceptation par client. En inventer une aurait fait passer une estimation arbitraire pour une vraie prédiction | Générer une probabilité par client à partir d'une méthode non vérifiée |
| Sur la page "Assistant IA", la courbe ROC et l'indicateur de dérive (PSI) sont recalculés en direct, en rejouant la même séparation des données à chaque fois (`seed=42`), avec un résultat gardé en mémoire le temps de la session (`st.cache_resource`), plutôt que d'afficher des valeurs fixes | Des chiffres figés une fois pour toutes deviennent obsolètes sans que personne ne s'en rende compte, dès que le modèle ou les données changent. Le recalcul est vérifiable et ne prend que quelques centaines de millisecondes | Afficher des chiffres codés en dur, ou recopiés d'un rapport ancien |
| Chaque page qui affiche des alertes montre aussi un signal positif réel, calculé à partir des données (portefeuille en bonne santé, opportunités identifiées), pas seulement des risques | Un tableau de bord qui ne montre que des problèmes donne une image faussée de la situation et n'aide pas à voir ce qui fonctionne bien | Se limiter aux seules alertes de risque |
