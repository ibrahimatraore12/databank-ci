# Design Decision Log - dataBank CI Customer 360

> *[Version française disponible : [decisions.md](decisions.md)]*

**Author:** Ibrahima TRAORÉ - Analytics Engineer

This document lists the important choices made during the project, why
they were made, and the other options that were not chosen.

Table format: Decision | Reason | Alternative rejected

| Decision | Reason | Alternative rejected |
|----------|--------|----------------------|
| The disengagement score is computed in two steps: explicit business rules in Step 1, then a Machine Learning experiment on an approximate indicator in Step 2 | The source file contains no actually observed customer departure (see `docs/ml_problem_definition_en.md`). Presenting a Machine Learning score as certain truth would have been misleading | Train a Machine Learning model directly, without a business-rules step |
| Generate a synthetic dataset (`is_synthetic=True`) to bring the volume from 140 to about 540 customers | With a single simple criterion, only 2 customers are affected (1.4%). Even with a broader criterion (25.0%), the sample stays small on 140 customers. A larger test sample is needed to compare models reliably | Train only on the 140 real customers and accept unreliable results |
| The threshold used to define an "at-risk" customer is set at 2 out of 4 criteria (not tuned to reach a target rate in advance) | Measured on real data: a single criterion flags 70% of customers (far too many), 2 criteria give 25.0%, 3 criteria give only 0.7% (too few). The 2-criteria threshold is the best compromise, kept as-is rather than adjusted afterward to hit a specific number | Search for a threshold or weighting that would artificially produce 12-15% of customers affected |
| All artificially generated data stays flagged `is_synthetic=True` in the final tables and in the dashboard | So no one can mistake generated data for real customer data | Silently mix real and synthetic data |
| The revenue generated per customer (NBI) is estimated with the standard WAEMU formula, and flagged `estimated_nbi_flag=True` | The real accounting NBI figure doesn't exist in the source file | Don't compute any NBI at all |
| The same starting value (`seed=42` and `random_state=42`) is used everywhere: dbt tests, the data split for Machine Learning, synthetic data generation | So that re-running the pipeline always produces exactly the same result | Let randomness vary on every run |
| No technical column name appears on the dashboard: everything goes through the `LABELS` file (`dashboard/components/ui.py`) and the `i18n/*.json` translation files | The dashboard is built for relationship managers, not technical staff | Display raw SQL or pandas column names directly |
| The payment delay threshold (`days_past_due`) is set at 15 days to correct a loan's status in `stg_loans.sql` | Observed during exploratory analysis: loans actually classified `Watchlist` or `Delinquent` exceed this threshold, `Current` loans stay below it | Use the source file's status directly, without checking it against the real payment delay |
| Files uploaded by a user are saved to a private Google Cloud Storage space, reloaded at every instance's startup (`src/storage_sync.py`) | Cloud Run doesn't keep anything permanently in memory: a file uploaded from the Administration tab would be lost on the very first restart without this external backup. Detail in `docs/architecture_en.md`, section 6 | Save only the source Excel file, or use a separately managed database |
| A schema-version check (`config.DATA_SCHEMA_VERSION`) runs before any download from the GCS storage space | Without this check, a future structural change in dbt could be silently overwritten at startup by an old restored backup, causing cascading errors on any query using the new column | Download without any check and accept the incompatibility risk |
| The `mlflow.db` file is cleanly copied using the `sqlite3.Connection.backup()` function to a temporary file before being sent to GCS, never copied directly byte for byte | MLflow can keep a connection open in the background; a raw copy would risk capturing the data at an inconsistent moment | Copy the `.db` file directly, without this clean backup step |
| The "Commercial Opportunities" page is built from real historical acceptance rates, by offer type (`main_staging.stg_offers`), rather than a predicted probability for each customer | No model in this project computes a per-customer acceptance probability. Inventing one would have passed off an arbitrary estimate as a real prediction | Generate a per-customer probability using an unverified method |
| On the "AI Assistant" page, the ROC curve and the drift indicator (PSI) are recomputed live, replaying the same data split every time (`seed=42`), with the result kept in memory for the session (`st.cache_resource`), instead of showing fixed values | Numbers frozen once and for all quietly become outdated as soon as the model or the data changes. The recomputation is verifiable and takes only a few hundred milliseconds | Show hard-coded numbers, or numbers copied from an old report |
| Every page that shows alerts also displays a real positive signal, computed from the data (healthy portfolio, identified opportunities), not just risks | A dashboard that only shows problems gives a distorted picture of the situation and doesn't help show what's working well | Limit the page to risk alerts only |
