# dataBank CI - Customer 360

> *[Version française disponible : [README.md](README.md)]*

This project analyzes dataBank CI's customers, from the first raw data file
all the way to the final dashboard. It includes: loading the data, data
quality checks, transformations with dbt, a customer churn risk score, a
Streamlit dashboard, and an MCP server that lets you ask questions in plain
language.

**Author:** Ibrahima TRAORÉ - Analytics Engineer
**Tools used:** Python · pyenv · dbt · DuckDB · MLflow · Streamlit · Docker

## How the data flows

```
starter_dataset.xlsx (10 tables)
        │
        ▼
   src/ingest.py ──► Bronze (DuckDB, real + synthetic data)
        │
        ▼
   dbt_project/models/staging      (10 models: typing and corrections)
        │
        ▼
   dbt_project/models/intermediate (calculations on customer behavior)
        │
        ▼
   dbt_project/models/marts        (Gold: customer_360, customer_segments, nba)
        │
        ├──► dashboard/  (Streamlit dashboard, 9 pages, FR/EN)
        ├──► mcp_server/ (MCP server, 5 read-only tools)
        └──► ml/         (rule-based score + compared Machine Learning models)
```

Note: "Bronze", "Silver" and "Gold" are the three classic stages of a data
pipeline. Bronze = raw data exactly as received. Silver = cleaned and typed
data (the `staging` folder). Gold = data that is ready to use, aggregated
and trustworthy (the `marts` folder), which is what the dashboard and the
MCP server read from.

The dashboard's 9 pages share the same visual style (black and orange,
Artefact charte) through shared building blocks defined once in
`dashboard/components/ui.py` (page banner, reading guide, section headers,
indicator cards with red/orange/green color coding, alert messages). No
page should have its own style that differs from the others.

cd databank-ci

## Running the full pipeline

```bash
# 1. Load raw data + enrichment + synthetic data generation
python3 pipelines/run_pipeline.py

# 2. Data transformation with dbt (Bronze -> Silver -> Gold)
cd dbt_project
export DBT_PROFILES_DIR=$(pwd)
dbt run
dbt test
cd ..

# 3. Machine Learning pipeline (rule-based score + model comparison + MLflow tracking)
python3 pipelines/run_ml_pipeline.py

# 4. Dashboard
streamlit run dashboard/APP.py

# 5. MCP server
python3 mcp_server/databank_mcp_server.py
```

## FAQ

**Why DuckDB instead of PostgreSQL?**
The dataset weighs less than 10 MB. DuckDB runs directly inside Docker
without needing a separate database server to operate. If the data volume
ever grows past 10 GB, switching to BigQuery just means changing the
`profiles.yml` file, without touching the rest of the code (see
`docs/architecture_en.md`).

**Can we trust the score of the model used in production?**
On real data (140 customers, 35 of whom left): the model scores 0.913 AUC.
This number is indicative only, because the sample is too small to be sure
the model would behave as well on many more customers. On data enriched
with synthetic examples (540 customers): the score rises to 0.944, which is
more reliable. This still isn't the top score in the comparison
(RandomForest and XGBoost reach 1.0). Those two models were deliberately
not chosen: a perfect score on synthetic customers generated by repeated
sampling (bootstrap) is a sign the model memorized the examples, rather than
learning general patterns. More detail in `docs/model_comparison_en.md`.

**How do we avoid the model "cheating" during training (data leakage)?**
The `preprocess_data(fit=True)` function, which adjusts value scales, is
applied only to the training data. The result of that adjustment is saved
to `ml/artifacts/preprocessor.pkl`, then reused as-is on the test data,
without recomputing it.

**What does the MCP server add?**
With the dashboard, you have to search for information page by page. The
MCP server lets you ask a question directly in plain language, for example
"Which VIP customers are at risk?", and get back a structured answer,
computed in real time from DuckDB.

## Updating the data

The source file can be replaced without any technical work, from the
dashboard's Administration tab (a password is required, see
`ADMIN_PASSWORD` in `docs/data_dictionary_en.md`).

**Steps to follow:**
1. Administration tab → "Load a new data file" section
2. Upload the Excel file (same format, 10 sheets)
3. Check the validation report (sheet by sheet)
4. If everything is valid, click "Recompute now"

The full pipeline runs again automatically (load → dbt → Machine Learning,
about 55 seconds measured). The result is visible immediately on this
instance, then saved to a private Google Cloud Storage bucket. This keeps
the data safe across restarts and shares it with every instance of the
app - dashboard and MCP server (AI Assistant) included, the latter being
refreshed right afterward. No further action is needed: no need to replace
a file in the code repository, and no need to rebuild a Docker image. The
full mechanism, the compatibility checks, and the known limitations are
explained in `docs/architecture_en.md` (section 6).

## Code rules to follow

 - Code comments are written in two languages (French then English) on
   points that aren't obvious to understand.
- No technical column name should ever appear in the dashboard: always go
  through `dashboard/components/ui.py::LABELS` and `dashboard/i18n/` for
  displayed labels.
- The pipeline can be run again and again without issue: the
  `pipelines/run_pipeline.py` script cleans up its previous results before
  recreating them, and the same starting value (`seed=42`) is used
  everywhere (dbt, Machine Learning, synthetic data generation) so results
  stay reproducible.
- Any artificially generated data is flagged `is_synthetic=True` and is
  never mixed with real data in the default views.
