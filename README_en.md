## Architecture

```
starter_dataset.xlsx (10 tables)
        │
        ▼
   src/ingest.py ──► Bronze (DuckDB, real + synthetic)
        │
        ▼
   dbt_project/models/staging      (10 models, typing + business corrections)
        │
        ▼
   dbt_project/models/intermediate (behavioral computations)
        │
        ▼
   dbt_project/models/marts        (Gold: customer_360, customer_segments, nba)
        │
        ├──► dashboard/  (Streamlit, 9 pages, FR/EN)
        ├──► mcp_server/ (MCP server, 5 read-only tools)
        └──► ml/         (rule-based score + compared ML models)
```

The dashboard's 9 pages share a single visual identity (black/orange
Artefact charte) through common components
(`dashboard/components/ui.py`: page banner, reading guide, section
headers, RAG-thresholded KPI cards, alerts) - never ad hoc styling per
page.

## Running the full pipeline

```bash
# 1. Bronze + enrichment + synthetic generation
python3 pipelines/run_pipeline.py

# 2. dbt transformation (Bronze -> Silver -> Gold)
cd dbt_project
export DBT_PROFILES_DIR=$(pwd)
dbt run
dbt test
cd ..

# 3. ML pipeline (rule-based score + model comparison + MLflow)
python3 pipelines/run_ml_pipeline.py

# 4. Dashboard
streamlit run dashboard/APP.py

# 5. MCP server
python3 mcp_server/databank_mcp_server.py
```

## FAQ

**Why DuckDB rather than PostgreSQL?**
Dataset < 10 MB, embedded in Docker, no server to operate. Above ~10 GB →
migrate to BigQuery by changing only `profiles.yml` (see
`docs/architecture_en.md`).

**Is the production model's AUC reliable?**
On real data (n=140, 35 positives): 0.913, indicative only - the sample is
too small to generalize from. On enriched data (n=540): 0.944, more robust.
It isn't the comparison's top score (RandomForest and XGBoost reach 1.0): I
didn't retain those two models precisely because a perfect score on
bootstrap synthetic customers is a memorization signal, not generalization.
Detail in `docs/model_comparison_en.md`.

**How did I avoid data leakage?**
`preprocess_data(fit=True)` on the training set only. The scaler is saved to
`ml/artifacts/preprocessor.pkl` and applied to the test set without
recomputation.

**What does the MCP server add?**
The dashboard requires searching page by page. The MCP server brings the
answer back in natural language: "Which VIP customers are at risk?" →
a structured answer, computed from DuckDB in real time.

## Updating the data

The source file can be reloaded without technical intervention from the
dashboard's Administration tab (password required, see `ADMIN_PASSWORD` in
`docs/data_dictionary_en.md`).

**Procedure:**
1. Administration tab → "Load a new data file" section
2. Upload the Excel file (same format, 10 sheets)
3. Check the validation report (sheet by sheet)
4. If valid, click "Recompute now"

The full pipeline replays (ingestion → dbt → ML, ~55 seconds measured), the
result is immediately visible on this instance, then saved to a private
Google Cloud Storage bucket to survive restarts and be picked up by every
instance - dashboard and MCP server (AI Assistant) included, the latter
being explicitly resynced right afterward. No further action is needed (no
file replacement in the repo, no image rebuild). Detail on the mechanism,
the schema-compatibility guard, and the accepted limitations in
`docs/architecture_en.md` (section 6).

## Code rules

 - Bilingual comments (French then English) on non-obvious points.
- Strict semantic layer: no technical column name in the dashboard, see
  `dashboard/components/ui.py::LABELS` and `dashboard/i18n/`.
- Idempotence: `pipelines/run_pipeline.py` cleans its outputs before
  recreating them; `seed=42` fixed everywhere (dbt, ML, synthetic generation).
- All generated data is flagged `is_synthetic=True` and never mixed with
  real data in the default views.
